# Feature Specification: Weekend Quest Planner

**Feature Branch**: `001-weekend-quest-planner`

**Created**: 2026-07-10

**Status**: Draft

**Input**: User description: "Weekend Quest Planner의 첫 번째 MVP 기능을 명세로 작성해주세요.

목표:
사용자가 오늘의 기분, 사용 가능 시간, 예산, 동행 여부를 선택하면
조건에 맞는 미니 코스를 추천하는 앱을 만듭니다.

사용자:
- 주말이나 퇴근 후에 무엇을 할지 정하지 못한 사람
- 돈과 시간을 많이 쓰지 않고 가벼운 활동을 찾고 싶은 사람
- 복잡한 일정 앱이 아니라 바로 실행 가능한 추천을 원하는 사람

MVP 범위:
- React 기반 단일 페이지 앱
- 외부 API, 로그인, 서버 없음
- mock data 기반 추천
- 최근 추천 결과는 localStorage에 저장
- 모바일과 데스크톱 모두 지원

첫 기능:
- 기분, 시간, 예산, 동행 여부 입력
- 조건에 맞는 코스 추천
- 각 활동의 예상 시간과 예상 비용 표시
- 조건에 맞는 추천이 없을 때 빈 상태 안내"

## Clarifications

### Session 2026-07-10

- Q: 사용자가 조건을 바꾸면 추천이 즉시 갱신되어야 하나요, 아니면 추천 버튼을 눌러야 하나요? → A:
  사용자가 조건을 모두 선택한 뒤 `추천하기` 버튼을 눌러 결과를 본다.
- Q: 최근 추천은 몇 건까지 저장해야 하나요? → A: 최근 추천은 가장 마지막 3건만 저장한다.
- Q: 추천 결과는 몇 개를 보여줘야 하나요? → A: 조건에 맞는 가장 적합한 코스 1개만 보여준다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 조건에 맞는 코스 찾기 (Priority: P1)

사용자는 기분, 사용 가능 시간, 예산, 동행 여부를 선택한 뒤 `추천하기` 버튼을 눌러 자신에게
맞는 미니 코스 1개를 찾을 수 있어야 합니다.

**Why this priority**: 이 기능이 있어야 앱의 핵심 가치가 성립합니다.

**Independent Test**: 사용자가 조건을 선택하고 `추천하기` 버튼을 눌렀을 때, 조건에 맞는 추천
코스 1개가 표시되는지 확인하면 독립적으로 검증할 수 있습니다.

**Acceptance Scenarios**:

1. **Given** 사용자가 아무 조건도 선택하지 않은 상태, **When** 기분, 시간, 예산, 동행 여부를
   선택하고 `추천하기`를 누르면, **Then** 조건에 맞는 미니 코스 1개가 표시된다.
2. **Given** 사용자가 특정 조건 조합을 선택한 상태, **When** `추천하기`를 눌렀을 때 해당 조건에
   맞는 코스가 존재하면, **Then** 결과 화면에는 가장 적합한 1개만 표시된다.

---

### User Story 2 - 추천 정보 확인하기 (Priority: P2)

사용자는 추천된 각 활동의 예상 시간과 예상 비용을 함께 확인할 수 있어야 합니다.

**Why this priority**: 시간과 비용 정보가 있어야 사용자가 바로 실행할 수 있는지 판단할 수
있습니다.

**Independent Test**: 추천 결과에서 각 항목의 시간과 비용이 모두 표시되는지 확인하면 독립적으로
검증할 수 있습니다.

**Acceptance Scenarios**:

1. **Given** 추천 결과가 표시된 상태, **When** 사용자가 각 활동을 살펴보면, **Then** 각 활동마다
   예상 시간과 예상 비용이 함께 보인다.
2. **Given** 예상 시간이 정확한 값이 아닌 범위로 표현되어야 하는 활동, **When** 결과가 표시되면,
   **Then** 사용자는 범위 또는 명확한 표기로 시간과 비용 정보를 이해할 수 있다.

---

### User Story 3 - 결과가 없을 때 안내 받기 (Priority: P3)

사용자는 현재 선택한 조건에 맞는 코스가 없을 때 다음 행동을 알 수 있는 안내를 받아야 합니다.

**Why this priority**: 추천 결과가 없을 때도 사용자가 앱을 이탈하지 않고 조건을 다시 조정할 수
있어야 합니다.

**Independent Test**: 조건을 일부러 좁게 선택해 결과가 없는 상태를 만들고, 빈 상태 안내가
표시되는지 확인하면 검증할 수 있습니다.

**Acceptance Scenarios**:

1. **Given** 선택한 조건에 맞는 코스가 없는 상태, **When** 추천을 요청하면, **Then** 결과 없음
   상태가 표시된다.
2. **Given** 결과 없음 상태, **When** 사용자가 조건을 다시 조정할 수 있는 화면을 보면, **Then**
   다음 시도에 필요한 안내를 확인할 수 있다.

---

### User Story 4 - 최근 추천 다시 보기 (Priority: P3)

사용자는 최근에 확인한 추천 결과를 다시 볼 수 있어야 합니다.

**Why this priority**: 이전 선택을 다시 찾을 수 있으면 반복 사용이 쉬워집니다.

**Independent Test**: 추천을 한 뒤 다시 앱을 열었을 때 최근 추천이 남아 있는지 확인하면
독립적으로 검증할 수 있습니다.

**Acceptance Scenarios**:

1. **Given** 사용자가 이전에 추천 결과를 확인한 상태, **When** 다시 앱을 열면, **Then** 최근
   추천 결과가 유지되어 보인다.
2. **Given** 최근 추천 결과가 저장된 상태, **When** 사용자가 다시 추천을 확인하면, **Then**
   이전 추천 기록을 참고할 수 있다.

### Edge Cases

- 선택 조건이 매우 제한적이어서 추천이 하나도 없을 때는 빈 상태 안내를 보여줘야 합니다.
- 사용자가 조건을 바꿨을 때 이전 결과가 현재 조건과 섞여 보이면 안 됩니다.
- 최근 추천이 없을 때도 화면이 어색하지 않게 기본 상태를 보여줘야 합니다.
- 모바일 화면에서도 입력 항목과 결과가 한 화면 안에서 읽기 쉽게 보여야 합니다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 사용자는 오늘의 기분, 사용 가능 시간, 예산, 동행 여부를 입력할 수 있어야 한다.
- **FR-002**: 시스템은 사용자가 조건을 모두 선택한 뒤 `추천하기`를 요청했을 때, 조건에 맞는
  미니 코스 1개를 추천해야 한다.
- **FR-003**: 시스템은 추천 결과에서 각 활동의 예상 시간과 예상 비용을 함께 표시해야 한다.
- **FR-004**: 시스템은 조건에 맞는 결과가 없을 때 빈 상태 안내를 표시해야 한다.
- **FR-005**: 시스템은 최근 추천 결과를 최대 3건까지 저장하고, 사용자가 다시 방문했을 때 이전에
  본 추천 결과를 확인할 수 있게 해야 한다.
- **FR-006**: 시스템은 추천이 사용자의 선택된 조건 중 어떤 항목과 일치했는지 간단히 보여줘야
  한다.
- **FR-007**: 시스템은 모바일과 데스크톱 환경 모두에서 읽기 쉽고 조작 가능한 화면을 제공해야 한다.

### Key Entities *(include if feature involves data)*

- **사용자 선택 조건**: 기분, 사용 가능 시간, 예산, 동행 여부로 구성되는 추천 기준.
- **미니 코스**: 조건에 따라 제안되는 활동 묶음으로, 예상 시간과 예상 비용을 포함한다.
- **최근 추천 기록**: 사용자가 최근에 확인한 추천 결과의 요약 정보.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 조건 입력 후 `추천하기` 버튼을 눌러 추천 결과 또는 빈 상태 안내를
  확인할 수 있어야 한다.
- **SC-002**: 추천 결과 1개에는 예상 시간과 예상 비용이 항상 함께 보여야 한다.
- **SC-003**: 결과가 없는 경우에도 사용자는 다음에 바꿀 조건을 이해할 수 있는 안내를 받아야 한다.
- **SC-004**: 최근 추천 기록은 앱을 다시 열었을 때도 확인 가능해야 한다.
- **SC-005**: 모바일과 데스크톱에서 주요 입력과 결과 확인이 가로 스크롤 없이 완료되어야 한다.

## Assumptions

- 사용자는 회원가입이나 로그인 없이 앱을 바로 사용할 수 있다고 가정합니다.
- 추천 데이터는 사전에 준비된 가벼운 활동 묶음으로 충분하다고 가정합니다.
- 최근 추천 기록은 개인 기기에서만 유지되는 수준이면 충분하다고 가정합니다.
- 초기 버전에서는 외부 서비스 연동이나 실시간 일정 정보는 필요하지 않다고 가정합니다.
