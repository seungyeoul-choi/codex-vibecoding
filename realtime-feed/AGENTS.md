# AGENTS.md

## 프로젝트 개요

이 저장소는 `Realtime Guestbook` 프로젝트를 위한 작업 공간입니다.

제품 목표:

- 방문자를 위한 따뜻한 실시간 방명록 경험 제공
- 포스트잇 스타일의 갤러리 기반 UI 구성
- 사진 또는 직접 그린 그림 첨부 지원
- 상세 화면과 댓글 UI 지원
- 이후 Supabase 기반 실시간 업데이트로 확장 가능하도록 설계

## 작업 원칙

### 디자인 원칙

- UI는 기술적인 피드보다 따뜻한 메모 보드처럼 느껴져야 합니다.
- 시각적으로는 정돈된 불완전함을 허용하되, 사용성은 명확해야 합니다.
- 과한 모션, 네온 감성, 대시보드형 실시간 표현은 피합니다.
- 모바일 우선 구조를 유지합니다.

### 제품 원칙

- 로그인 없이 빠르게 참여할 수 있어야 합니다.
- 작성 성공 후 사용자는 즉시 반영된 결과를 체감할 수 있어야 합니다.
- 폴링 또는 향후 실시간 갱신 실패 시에도 기존 목록은 유지되어야 합니다.
- MVP 범위 밖 기능은 임의로 추가하지 않습니다.

### 구현 원칙

- 서버와 클라이언트의 검증 규칙은 일관되어야 합니다.
- 데이터 접근은 중앙화된 레이어 뒤로 숨깁니다.
- UI 구현 시 디자인 문서와 스펙 문서가 충돌하면 먼저 스펙을 확인하고, 사용자 경험 관련 결정은 디자인 문서 기준으로 정리합니다.
- 새로운 화면이나 컴포넌트를 추가할 때는 기존 문서에 없는 패턴을 임의로 확장하지 말고, 먼저 문서에 맞는지 확인합니다.

## 문서 우선순위

문서 간 판단 우선순위는 아래와 같습니다.

1. 제품 요구사항과 동작 정의: 스펙 문서
2. UI/UX 방향과 화면 패턴: 디자인 문서
3. 구현 순서와 파일 구조: 구현 계획 문서

## 에이전트 체크리스트

작업 시작 전:

- `DESIGN.md`를 읽었는가
- 스펙 문서의 MVP 범위를 확인했는가
- 구현 계획 문서의 대상 파일과 순서를 확인했는가

UI 작업 전:

- 포스트잇 갤러리, 작성 폼, 상세 화면 패턴이 디자인 문서와 일치하는가
- 모바일 우선순위를 해치지 않는가
- 빈 상태, 로딩, 에러, 연결 상태를 함께 고려했는가

기능 작업 전:

- 유효성 검증 규칙이 스펙 문서와 일치하는가
- 성공/실패 시 사용자 흐름이 정의된 기준과 맞는가
- 실시간 또는 폴링 실패 시 복원력이 유지되는가
