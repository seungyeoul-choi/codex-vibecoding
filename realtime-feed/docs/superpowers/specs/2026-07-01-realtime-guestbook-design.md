## 1. Executive Summary

### Problem Statement

로그인 없이도 방문자가 짧은 메시지를 남기고 최근 메시지를 바로 확인할 수 있는 가벼운 방명록 경험이 필요하다. 초기 버전에서는 구현 복잡도를 낮추면서도 새로고침 없이 최근 등록 내역이 반영되는 체감형 실시간성이 핵심이다.

### Proposed Solution

Next.js App Router, 커스텀 API Route, MySQL 기반의 단순한 방명록 웹 앱을 구축한다. 사용자는 닉네임과 메시지를 입력해 글을 등록할 수 있고, 목록은 3초에서 5초 간격의 폴링으로 자동 갱신되어 별도 새로고침 없이 최근 항목을 확인할 수 있다.

### Success Criteria

- 유효한 입력으로 등록된 방명록 엔트리는 `POST /api/guestbook` 요청 성공 후 즉시 저장되고, 동일 세션 UI에서 1회 추가 조회 이내에 목록에 반영된다.
- 방명록 목록 조회는 기본 정렬 기준 `created_at DESC`를 만족하며 최신 50개 항목 이내를 반환한다.
- 닉네임과 메시지의 유효성 검증 실패 시 서버는 `400` 응답을 반환하고, 클라이언트는 사용자가 수정 가능한 형태의 오류 상태를 유지한다.
- 폴링 갱신 실패 시 기존 목록은 유지되며, 다음 주기의 재시도로 서비스 이용이 계속 가능해야 한다.
- MVP 범위 내 핵심 API 및 UI 흐름 테스트가 자동화되어야 하며, 최소 항목은 등록 성공, 등록 실패, 빈 상태, 최신순 정렬, 자동 갱신이다.

## 2. User Experience & Functionality

### User Personas

- 방문자: 로그인 없이 빠르게 닉네임과 메시지를 남기고 싶어 하는 일반 사용자
- 운영자/개발자: 구현 범위가 명확하고, 이후 스팸 방지나 관리 기능을 확장하기 쉬운 구조를 원하는 팀

### Goals

- 방문자가 닉네임과 메시지로 방명록 항목을 작성할 수 있어야 한다.
- 방명록 데이터는 MySQL에 저장되어 새로고침이나 서버 재시작 이후에도 유지되어야 한다.
- 최근 항목은 닉네임, 메시지, 작성 시각과 함께 깔끔한 목록으로 표시되어야 한다.
- 사용자는 전체 페이지 새로고침 없이 최근 항목을 자동으로 확인할 수 있어야 한다.
- 첫 버전은 작고 단순하게 유지하여 이후 확장이 쉬워야 한다.

### User Stories

- 방문자로서, 로그인 없이 닉네임과 메시지를 남기고 싶다. 그래서 빠르게 참여할 수 있다.
- 방문자로서, 내가 작성한 메시지가 곧바로 목록에 나타나길 원한다. 그래서 등록이 정상 처리되었는지 바로 확인할 수 있다.
- 방문자로서, 다른 사용자의 최근 메시지를 자동으로 보고 싶다. 그래서 직접 새로고침하지 않고도 최신 상태를 확인할 수 있다.
- 개발자로서, 데이터 접근과 유효성 검증이 일관된 구조이길 원한다. 그래서 유지보수와 기능 확장이 쉬워진다.

### Acceptance Criteria

- 사용자가 방명록 페이지에 진입하면 입력 폼과 최신 항목 목록이 표시된다.
- 닉네임은 trim 이후 2자 이상 20자 이하일 때만 유효하다.
- 메시지는 trim 이후 1자 이상 300자 이하일 때만 유효하다.
- 공백만 포함한 입력은 클라이언트와 서버 모두에서 거부된다.
- 사용자가 유효한 값을 제출하면 중복 제출이 방지되고, 성공 시 폼은 초기화된다.
- 신규 등록 성공 후 목록은 즉시 다시 조회되어 최신 항목이 반영된다.
- 목록은 기본적으로 `created_at DESC` 순서로 보여야 한다.
- 항목이 없을 때는 친화적인 빈 상태 메시지를 표시해야 한다.
- 폴링 요청 실패 시 화면의 기존 항목은 사라지지 않아야 하며, 다음 주기에 자동 재시도되어야 한다.

### User Flow

1. 방문자가 방명록 페이지를 연다.
2. 페이지는 최근 항목 목록과 입력 폼을 보여준다.
3. 방문자가 닉네임과 메시지를 입력하고 제출한다.
4. 클라이언트는 필수값과 길이 제한을 먼저 검증한다.
5. 서버는 동일 규칙으로 다시 검증하고, 유효하면 MySQL에 저장한다.
6. 저장 성공 시 폼은 초기화되고, 짧은 성공 상태를 보여준 뒤 목록을 즉시 갱신한다.
7. 이후 페이지는 3초에서 5초 간격으로 목록을 다시 조회한다.
8. 등록된 항목이 하나도 없으면 빈 상태 메시지를 보여준다.

### Non-Goals

- 인증 또는 사용자 계정
- 항목 수정 또는 삭제
- 좋아요, 답글, 스레드형 댓글
- CAPTCHA 및 고도화된 스팸 방지
- WebSocket 또는 SSE 기반의 진정한 푸시형 실시간 처리

## 4. Technical Specifications

### Architecture Overview

애플리케이션은 Next.js App Router를 기반으로 UI와 API를 함께 제공한다. 화면은 크게 입력 폼과 방명록 목록으로 구성되며, 프론트엔드는 내부 API Route를 통해 데이터 생성과 조회를 수행한다.

백엔드는 요청을 검증한 뒤 MySQL에 기록하고, 프론트엔드가 사용하기 쉬운 정규화된 응답을 반환한다. 목록은 초기 진입 시 1회 조회되고, 이후 3초에서 5초 간격의 고정 폴링으로 다시 조회된다. 이 방식은 영속 연결 없이도 체감형 실시간성을 제공하며 MVP 복잡도를 낮춘다.

MySQL은 단일 소스 오브 트루스로 동작한다. 데이터 접근은 공통 데이터베이스 레이어를 통해 수행하며, 나머지 코드가 직접 쿼리에 결합되지 않도록 한다. 본 프로젝트 규모에서는 Prisma를 기본 선택지로 사용해 스키마 관리, 타입 안정성, 데이터 접근 단순성을 확보한다.

### Components

#### Frontend

- `GuestbookPage`: 목록 로딩, 폼 제출, 즉시 갱신, 주기적 폴링을 조율하는 최상위 페이지
- `GuestbookForm`: 닉네임과 메시지를 입력받고 클라이언트 검증 후 API로 제출하는 폼
- `GuestbookList`: 최신순 항목 목록 또는 빈 상태를 렌더링하는 목록 컴포넌트
- `GuestbookEntry`: 닉네임, 메시지, 작성 시각을 표시하는 단일 항목 컴포넌트

#### Backend

- `GET /api/guestbook`: 최신 방명록 항목을 역순으로 조회해 반환
- `POST /api/guestbook`: 입력값을 검증하고 새 방명록 항목을 생성
- Database access module: Prisma 기반 MySQL 접근을 중앙화하는 공통 모듈

### Data Model

초기 버전에서는 단일 테이블만 사용한다.

#### Table: `guestbook_entries`

- `id`: 기본 키
- `nickname`: 화면 표시에 사용하는 필수 문자열
- `message`: 필수 메시지 본문
- `created_at`: 정렬과 표시에 사용하는 생성 시각

### Validation Rules

- `nickname`: trim 이후 2자 이상 20자 이하
- `message`: trim 이후 1자 이상 300자 이하
- 공백만 포함한 값은 거부

### Query Rules

- 기본 목록 정렬은 `created_at DESC`
- 초기 버전 반환 범위는 최신 50개 항목 이내

### Integration Points

- Frontend ↔ Backend: Next.js App Router Route Handler 기반 내부 API 통신
- Backend ↔ Database: Prisma를 통한 MySQL CRUD
- 배포 및 런타임 환경 변수: `DATABASE_URL` 필수

### Security & Privacy

- 서버 검증을 최종 권위로 둔다.
- 내부 데이터베이스 오류는 사용자에게 직접 노출하지 않는다.
- 초기 버전에서는 익명 작성만 허용하며, 개인정보 수집 범위는 닉네임과 메시지 본문으로 제한한다.
- 입력값은 trim 후 저장해 불필요한 데이터 편차를 줄인다.
- 고급 스팸 방지, 속도 제한, 관리자 제어 기능은 v1 범위 밖으로 둔다.

### Error Handling And Validation

#### Client-side behavior

- 빈 값, 길이 초과, 공백-only 입력에 대해 즉시 오류를 보여준다.
- 제출이 성공하기 전까지 사용자가 입력한 값은 유지된다.
- 요청 진행 중에는 중복 제출을 비활성화한다.

#### Server-side behavior

- 유효하지 않은 요청은 `400`으로 거부하고 명확한 오류 정보를 반환한다.
- 예기치 않은 실패는 `500`과 일반화된 메시지로 응답한다.
- 내부 예외 세부사항은 사용자 응답에 포함하지 않는다.

#### Polling behavior

- 폴링 실패 시 현재 표시 중인 목록은 유지한다.
- 실패 후 다음 폴링 주기에서 자동 재시도한다.

#### Basic abuse prevention

- 입력값은 trim 및 정규화 후 검증/저장한다.
- 고도화된 안티스팸은 MVP 범위에서 제외한다.
- 운영 중 악용 징후가 보이면 가벼운 클라이언트 단위 쿨다운을 후속 검토한다.

### Testing

#### API tests

- 유효한 방명록 등록 요청이 성공한다.
- 빈 닉네임은 거부된다.
- 빈 메시지는 거부된다.
- 제한을 초과한 닉네임은 거부된다.
- 제한을 초과한 메시지는 거부된다.
- 목록은 최신순으로 반환된다.

#### UI tests

- 폼이 정상 렌더링되고 입력을 받는다.
- 잘못된 입력에 대해 클라이언트 검증 오류가 표시된다.
- 제출 성공 시 폼이 초기화된다.
- 항목이 없을 때 빈 상태가 표시된다.

#### Integration checks

- 항목 등록 후 목록이 즉시 갱신되어 새 메시지가 보인다.
- 폴링으로 전체 페이지 새로고침 없이 새 항목이 반영된다.
- 폴링 실패가 현재 목록을 지우지 않는다.

초기 구현에서는 핵심 API 검증과 등록 성공/실패 UI 흐름에 대한 자동화 테스트를 우선한다. 그 외 검증은 제품 규모가 커지기 전까지 수동 점검으로 보완할 수 있다.

### Implementation Notes

- 방명록 API는 App Router Route Handler를 우선 사용한다.
- 데이터 접근은 단일 모듈 뒤로 숨겨 쿼리 분산을 막는다.
- 실시간 전송 대신 단순한 클라이언트 폴링으로 MVP를 구현한다.
- 초기 릴리스는 시각적 화려함보다 안정성과 단순성을 우선한다.

## 5. Risks & Roadmap

### Phased Rollout

#### MVP

- 익명 닉네임 + 메시지 등록
- 최신순 목록 조회
- 3초에서 5초 간격 폴링 갱신
- 기본 유효성 검증
- 핵심 API/UI 자동화 테스트

#### v1.1

- 경량 속도 제한 또는 클라이언트 쿨다운
- 빈번한 실패 상황에 대한 운영 가시성 개선
- 목록 표시 품질 개선

#### v2.0

- 관리자 전용 삭제 기능
- 페이지네이션 또는 무한 스크롤
- WebSocket/SSE 기반 푸시형 실시간 전환 검토
- 확장된 입력 표현력(예: 이모지 또는 서식 일부 지원)

### Technical Risks

- 폴링 주기가 짧아질수록 사용자 수 증가 시 불필요한 조회 비용이 커질 수 있다.
- 익명 입력 구조상 스팸 또는 악의적 메시지 유입 위험이 있다.
- MySQL 연결 설정이 불안정하면 등록 및 목록 조회 모두 동시에 영향을 받는다.
- 서버와 클라이언트 검증 규칙이 분리되면 동작 불일치가 발생할 수 있으므로 공통 스키마 유지가 중요하다.
- 작성 시각 표시 포맷은 런타임 로캘에 따라 달라질 수 있어 향후 표시 기준 통일이 필요할 수 있다.

- 리치 텍스트 또는 이모지 지원
