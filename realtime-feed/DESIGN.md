# Realtime Guestbook 디자인 문서

## 문서 메타

- 문서 목적: Realtime Guestbook MVP의 UI/UX 방향성과 디자인 시스템 기준 정의
- 1차 독자: 프로덕트 디자이너, 프론트엔드 개발자
- 문서 범위: 방명록 작성 흐름, 포스트잇 갤러리, 상세 화면, 댓글, 시스템 상태, 모바일 우선 레이아웃
- 문서 상태: 구현 핸드오프를 위한 디자인 방향 문서

## 1. 제품 의도

Realtime Guestbook은 방문자가 짧은 메시지를 남기고, 그 메모가 살아 있는 공동 보드에 바로 붙는 느낌을 받는 따뜻하고 가벼운 방명록입니다.

이 제품은 다음과 같은 인상을 주어야 합니다.

- 기술적이기보다 따뜻하고 환영하는 느낌
- 복잡하기보다 가볍고 즉각적인 느낌
- 기업형 서비스처럼 딱딱하기보다 사람의 흔적이 느껴지는 분위기
- 과장되지 않은 실시간 반응으로 조용히 살아 있는 느낌

핵심 비유는 사진, 손그림, 짧은 메모가 함께 쌓이는 디지털 포스트잇 메모 보드입니다.

## 2. 경험 원칙

### 2.1 따뜻한 첫인상

- 첫 화면은 부드럽고 초대하는 느낌을 줘야 합니다.
- 누구나 쉽게 메모를 남길 수 있을 것처럼 보여야 합니다.
- 기능성보다 환대의 분위기가 먼저 전달되어야 합니다.

### 2.2 빠른 참여

- 방명록 작성은 페이지에서 가장 쉬운 행동이어야 합니다.
- 작성 영역은 초반에 잘 보이는 위치에 있어야 합니다.
- 검증 메시지는 차갑거나 기술적인 문장보다 친절한 안내처럼 보여야 합니다.

### 2.3 절제된 실시간성

- 실시간 반영은 새로운 메모가 보드에 붙는 느낌으로 표현해야 합니다.
- 피드백은 분명하되 조용해야 합니다.
- 화면이 활동 모니터링 대시보드처럼 보여서는 안 됩니다.

### 2.4 정돈된 불완전함

- 카드는 약간의 회전, 높이 차이, 개성 있는 배치가 가능해야 합니다.
- 동시에 간격, 타이포그래피, 인터랙션은 일관되고 예측 가능해야 합니다.
- 손으로 만든 느낌은 있어도 지저분해서는 안 됩니다.

## 3. 권장 디자인 방향

### 3.1 테마

`따뜻한 스크랩북 실시간 방명록`

### 3.2 스타일 조합

기본 방향:

- 접근성을 우선하는 명확한 UI 구조
- 빠르고 읽기 쉬운 플랫 기반 레이아웃
- 종이, 게시판, 포스트잇에서 온 부드러운 촉감적 인상

보조 감성:

- 가벼운 스크랩북 무드
- 카드와 컨트롤에 소량의 말랑한 입체감
- 소셜 피드의 강한 에너지보다 웰니스에 가까운 안정감

### 3.3 피해야 할 방향

- 과한 글래스모피즘
- 네온 중심의 사이버 감성
- 대시보드형 실시간 UI
- 지나치게 유아적인 스티커북 스타일
- 장식용 애니메이션 남발

## 5. 컬러 팔레트

### 5.1 팔레트 전략

- 메인 배경은 종이 같은 뉴트럴 톤을 사용합니다.
- 갤러리 카드는 파스텔 포스트잇 색으로 다양성을 줍니다.
- 주요 액션은 하나의 안정된 색으로 유지합니다.
- 실시간 연결과 상태는 별도의 상쾌한 강조색으로 표현합니다.
- 본문 텍스트 색은 일관되게 유지해 가독성을 확보합니다.

### 5.2 기본 팔레트

- `Paper`: `#FFFBF2`
- `Canvas`: `#F6F1E8`
- `Ink`: `#2F241F`
- `Soft Brown`: `#8A6F63`
- `Border`: `#E6D8C8`

### 5.3 포스트잇 팔레트

- `Butter Yellow`: `#F7E27B`
- `Peach`: `#F6C7A8`
- `Blush Pink`: `#EFB7C8`
- `Mint`: `#BFE3C5`
- `Sky`: `#B9D7F0`
- `Lavender`: `#D8C8F0`

### 5.4 액션 및 상태 팔레트

- `Primary Action`: `#3D7BE0`
- `Realtime Accent`: `#4FB7A8`
- `Success`: `#5FA36A`
- `Warning`: `#D9A441`
- `Error`: `#C96B5C`
- `Offline`: `#8C98A8`

### 5.5 컬러 사용 규칙

- 한 화면 안에서는 포스트잇 색을 `3~5개` 범위에서 반복 사용합니다.
- 카드 배경색을 CTA 색으로 재사용하지 않습니다.
- 본문 텍스트는 가능한 한 `Ink` 계열로 고정합니다.
- 에러 상태는 공격적이기보다 차분하고 도움되는 인상이어야 합니다.
- 실시간 상태 색은 형광색보다 민트나 맑은 블루 계열이 적합합니다.

## 6. 타이포그래피 방향

### 6.1 전체 원칙

- 제목은 감정이 느껴지고 약간 에디토리얼한 인상을 주어야 합니다.
- 본문은 읽기 리듬과 가독성을 가장 우선합니다.
- 손글씨 느낌은 포인트 수준으로만 사용하고, 핵심 읽기 텍스트에는 적용하지 않습니다.

### 6.2 추천 조합

우선 추천:

- `Lora` + `Raleway`
- `Playfair Display` + `DM Sans`

조금 더 캐주얼한 대안:

- `Fredoka` + `Nunito`

### 6.3 사용 규칙

- 히어로와 섹션 제목은 감성적인 서체를 사용합니다.
- 카드 본문, 댓글, 폼 라벨, 도움말은 실용적인 산세리프를 사용합니다.
- 스크립트 계열 폰트는 메시지 본문과 댓글에 사용하지 않습니다.
- 모바일 본문 최소 크기는 `16px`로 유지합니다.
- 본문 줄간격은 `1.5~1.65` 범위를 권장합니다.
- 시간 정보나 보조 라벨은 지나치게 작게 줄이지 말고 색상과 굵기로 우선순위를 조정합니다.

## 7. 간격과 모서리 반경 시스템

### 7.1 간격 스케일

- `4`: 미세 간격
- `8`: 아이콘과 텍스트 간격
- `12`: 조밀한 그룹 간격
- `16`: 기본 컨트롤 패딩
- `20`: 폼 그룹 간격
- `24`: 기본 카드 패딩
- `32`: 섹션 간격
- `40`: 카드 클러스터 간격
- `56`: 주요 페이지 섹션 간격

### 7.2 모서리 반경 규칙

- `8px`: 작은 컨트롤
- `14px`: 버튼, 입력창, 칩
- `18px`: 기본 카드
- `24px`: 모달, 상세 화면
- `16px`: 댓글 말풍선

### 7.3 표면 규칙

- 포스트잇 카드는 부드러워 보여야 하지만 과하게 통통하면 안 됩니다.
- 카드에는 약간의 회전이나 비대칭이 가능하지만 읽기와 클릭성이 우선입니다.
- 버튼과 입력창은 친근해야 하지만 장난감처럼 보여서는 안 됩니다.

### 7.4 그림자 규칙

- 기본 카드는 얕은 종이 그림자를 사용합니다.
- 인터랙션 시에는 과한 확대보다 그림자와 채도 변화가 우선입니다.
- 상세 화면은 갤러리 카드보다 더 또렷하게 배경에서 분리되어야 합니다.

## 8. 레이아웃 시스템

### 8.1 페이지 구조

권장 순서:

1. 간결한 헤더
2. 짧은 환영 문구 또는 제품 설명
3. 방명록 작성 영역
4. 포스트잇 갤러리
5. 필요 시 열리는 상세 화면

### 8.2 시각적 우선순위

- 작성 영역은 가장 중요한 행동이므로 갤러리보다 위에 위치해야 합니다.
- 갤러리는 제품의 감정적 중심입니다.
- 상세 화면은 읽기와 댓글을 깊게 만드는 보조 무대입니다.
- 실시간 연결 상태는 보이되 화면의 주인공이 되어서는 안 됩니다.

### 8.3 모션 규칙

- 한 화면에서 핵심 모션은 `1~2개` 정도로 제한합니다.
- 마이크로 인터랙션은 `150~300ms` 범위를 사용합니다.
- `prefers-reduced-motion`를 존중합니다.
- 무한 반복 장식 모션은 지양합니다.
- 애니메이션은 가능한 한 opacity와 transform 중심으로 구성합니다.

## 9. 방명록 작성 폼 패턴

### 9.1 패턴 이름

`Pinned Composer`

### 9.2 목적

작성 영역은 새 종이 한 장이 보드에 핀으로 꽂혀 있는 느낌이어야 하며, 방문자가 부담 없이 메모를 남길 수 있어야 합니다.

### 9.3 구조

- 닉네임 입력 필드
- 메시지 textarea
- 첨부 진입 영역
- 주요 제출 버튼
- 선택적 미리보기 영역

### 9.4 첨부 모델

두 가지 첨부 방식을 지원합니다.

- 사진 첨부
- 직접 그린 그림 첨부

권장 UI:

- 작은 세그먼트 토글 또는 두 개의 첨부 액션
- 하나의 첨부 슬롯과 미리보기
- 교체 및 삭제가 명확한 컨트롤

### 9.5 동작 규칙

- 폼은 단순하고 환영하는 인상이어야 합니다.
- 검증 실패 시 사용자가 입력한 내용은 유지합니다.
- 검증 메시지는 짧고 친절해야 합니다.
- 제출 중에는 중복 제출을 막습니다.
- 성공 피드백은 토스트에만 의존하지 말고, 새 메모가 갤러리에 나타나는 제품 고유의 반응을 우선합니다.

### 9.6 검증 문구 톤

예시:

- `닉네임은 2자 이상 입력해 주세요.`
- `메시지를 남겨 주세요.`
- `메시지는 300자 이내로 작성해 주세요.`

## 10. 포스트잇 카드 갤러리 패턴

### 10.1 패턴 이름

`Masonry Memory Wall`

### 10.2 목적

갤러리는 서로 다른 사람의 메모가 모여 개인적이면서도 읽기 쉬운 공용 보드처럼 보여야 합니다.

### 10.3 카드 구성

각 카드는 다음 요소를 포함합니다.

- 닉네임
- 상대 시간 또는 간단한 시간 표시
- 메시지 본문
- 선택적 이미지 또는 그림 미리보기
- 댓글 수 또는 상세 보기 액션

### 10.4 시각 규칙

- 메이슨리 느낌 또는 시각적으로 살짝 어긋난 그리드를 사용합니다.
- 카드 높이는 내용에 따라 자연스럽게 달라질 수 있습니다.
- 회전 값은 아주 작게만 사용합니다.
- 배경색이 달라도 카드 구조는 일관되어야 합니다.
- 테이프, 핀, 종이결 같은 장식은 아주 절제해서 사용합니다.

### 10.5 인터랙션 규칙

- 탭과 클릭으로 상세 화면이 열린다는 점이 명확해야 합니다.
- hover는 보조 강조용으로만 쓰고, 핵심 동작은 hover 없이 가능해야 합니다.
- 카드 리프트나 바운스 효과는 과하지 않게 유지합니다.
- 새 카드는 차분한 `fade + settle` 방식으로 들어오는 것이 적합합니다.

### 10.6 실시간 반영 규칙

- 새 카드는 방금 보드에 붙여진 메모처럼 보여야 합니다.
- 실시간 업데이트는 레이아웃 점프를 최소화해야 합니다.
- 필요하다면 짧게만 보이는 `방금 도착` 배지를 사용할 수 있습니다.

## 11. 상세 화면과 댓글 패턴

### 11.1 패턴 이름

`Letter View`

### 11.2 목적

상세 화면은 보드에서 메모 하나를 들어 올려 천천히 읽고, 그 아래에 달린 이야기를 보는 경험이어야 합니다.

### 11.3 상세 구조

- 확대된 카드 표면
- 전체 메시지 본문
- 첨부 사진 또는 그림
- 댓글 목록
- 댓글 작성 영역

### 11.4 반응형 표현 방식

- 모바일: 전체 화면 상세 보기 우선
- 데스크톱: 중앙 모달 또는 측면 패널

### 11.5 댓글 UI 규칙

- 댓글은 메인 카드보다 시각적으로 조용해야 합니다.
- 댓글에는 화려한 포스트잇 색보다 중립적인 표면을 사용합니다.
- 메인 포스트가 항상 주인공이어야 합니다.
- 댓글 작성자는 이름과 이니셜 배지 정도의 가벼운 식별로 충분합니다.
- 댓글 작성창은 상세 화면 하단 흐름과 가깝게 배치합니다.

### 11.6 첨부 표현 규칙

- 사진과 그림은 별도 미디어 피드가 아니라 메모의 일부처럼 보여야 합니다.
- 필요하다면 탭 시 더 크게 보는 집중 뷰를 제공합니다.

## 12. 상태 설계

### 12.1 빈 상태

목표:

- 첫 번째 참여를 자연스럽게 유도합니다.

가이드:

- 비어 있는 메모 보드 위에 안내용 카드 한 장이 보이는 구성이 적합합니다.
- 빈 상태 문구와 작성 영역 진입이 자연스럽게 연결되어야 합니다.
- 사과하는 말투보다 초대하는 말투를 사용합니다.

문구 예시 방향:

- `첫 메모를 남겨 이 보드를 시작해 보세요.`

### 12.2 로딩 상태

목표:

- 비동기 작업 중에도 화면이 살아 있다는 느낌을 유지합니다.

가이드:

- 빈 화면을 보여주지 않습니다.
- 갤러리에는 포스트잇 형태의 스켈레톤 카드를 사용합니다.
- 작성 폼은 가능한 한 먼저 보이게 유지합니다.
- 로딩 모션은 절제하고, reduced motion 환경을 고려합니다.

### 12.3 에러 상태

목표:

- 신뢰를 유지하고 흐름을 끊지 않습니다.

가이드:

- 목록 갱신 실패 시 기존 카드는 그대로 유지합니다.
- 부드러운 인라인 메시지 또는 배너를 사용합니다.
- 가능하면 `다시 시도` 액션을 제공합니다.
- 앱 전체가 내려간 상황이 아니라면 치명적 에러 화면처럼 보이지 않게 합니다.

문구 예시 방향:

- `불러오지 못했어요. 잠시 후 다시 시도할게요.`

### 12.4 실시간 연결 상태

목표:

- 연결 상태를 알리되, 시스템 모니터처럼 보이지 않게 합니다.

가이드:

- 작지만 눈에 띄는 상태 pill 또는 배너를 사용합니다.
- 문구는 사람 중심으로 차분하게 작성합니다.
- 연결이 불안정해도 기존 목록과 작성 기능은 계속 사용 가능해야 합니다.

상태 문구 예시:

- `실시간 연결 원활`
- `잠시 연결이 불안정해요`
- `다시 연결 중`

## 13. 모바일 레이아웃 우선순위

### 13.1 우선순위

1. 새 방명록 작성
2. 최신 메모 빠른 확인
3. 메모 상세 읽기와 댓글 작성
4. 사진 또는 그림 첨부의 명확한 사용성
5. 실시간 연결 상태의 빠른 이해

### 13.2 모바일 레이아웃 규칙

- 기본은 1열 카드 레이아웃을 사용합니다.
- 작성 영역은 화면 초반에 배치합니다.
- 엄지 손가락 기준으로 누르기 쉬운 간격과 액션 위치를 유지합니다.
- hover 전제를 두지 않습니다.
- 터치 타깃은 최소 `44x44`를 유지합니다.
- 고정 요소는 최소화해 작성 흐름을 방해하지 않게 합니다.

### 13.3 모바일 상세 화면

- 전체 화면 상세 뷰를 우선합니다.
- 가능하면 댓글 입력창은 하단 흐름에 가깝게 유지합니다.
- 첨부 미디어와 댓글이 서로 너무 멀어지지 않도록 균형을 맞춥니다.

## 14. 접근성 및 UX 가드레일

- 텍스트와 액션은 충분한 대비를 유지합니다.
- 키보드 탐색과 명확한 포커스 상태를 지원합니다.
- `prefers-reduced-motion`를 존중합니다.
- 중요한 상태 전달을 색상만으로 표현하지 않습니다.
- 모바일 상세 화면에서도 뒤로 가기 동작은 예측 가능해야 합니다.
- overflow 처리로 메모, 댓글, 첨부가 잘리지 않도록 주의합니다.

## 15. 향후 구현을 위한 메모

- 이후 구현에서 실시간 방식이 폴링에서 Supabase Realtime으로 바뀌더라도, 제품의 따뜻한 성격은 유지되어야 합니다.
- 실시간 기술은 신선함을 더하는 역할이어야 하며, 화면을 채팅 앱이나 피드처럼 바꾸면 안 됩니다.
- 기능이 늘어나더라도 댓글과 첨부는 여전히 하나의 메모 중심 경험 안에 묶여 보여야 합니다.
